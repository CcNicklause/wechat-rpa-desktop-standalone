关键防御：屏幕状态判定 (S-1/S-2)

为了防止 UI 漂移以及误操作，每次搜索后都会捕获微信当前截图，然后通过基于 WinRT OCR 的文字识别提取全页面词块进行模糊匹配判定。

1
极速判定已是好友 出现“发消息”或“音视频通话”等特征词直接置为 ALREADY_FRIEND 状态。
2
精准拦截账号不存在 通过“该用户不存在”等词命中 TARGET_NOT_FOUND 状态，直接阻断以防误加。
3
风控熔断保护 发现“操作频繁”或“账号异常”即刻置为 RISK_CONTROL 状态，开启全天任务熔断保护设备。

原生按键稳定性设计

旧版通过 pyautogui 高频模拟组合键容易导致 modifier key 粘滞卡死微信 UI 线程。升级方案采用 Windows 原生 UIAutomation 驱动：

  • 使用 SendKeys("{Ctrl}a{BackSpace}") 完成原生无感全选与删除。
  • 使用 SendKeys("{Ctrl}v") 实现无剪贴板残留的安全粘贴。
  • 在清空输入框 and 粘贴动作之间维持 0.5秒 的安全渲染延时。

上报信箱 (Outbox Pattern) 优势

为防止本地状态更新后，由于突然断网、服务器宕机或上游不可用导致的状态上报断裂，系统在 `demo.db` 中实现了两个物理发信箱表：

  • lead_status_reports：记录每个线索的业务中间态（如未找到用户、熔断、频控）
  • friend_check_reports：记录每个加友复查事件的最终对账结果（通过/拒绝）

独立的发信协程负责扫描这俩表，上报成功即标记为 `SENT`。这不仅实现了前后台服务解耦，更提供了 100% 可达的事件同步担保。

已识别的主要风险清单与闭环建议

P0 级别 本地管理接口未鉴权风险 (R=83)

现象： 前端调用上游日志 SSE 模块直接创建 EventSource 连接，请求头部不包含 Bearer Token，后端服务上相应路由亦缺乏 require_auth 校验。

闭环建议： 将 EventSource 接口替换为带有认证头部的 Fetch 缓冲流，并在后台强制对所有 /api/v1/upstream/* 节点增加鉴权依赖。

P0 级别 端口绑定冲突与强杀行为 (R=78)

现象： 桌面前端将 API 服务地址硬编码为固定 8000 端口。在打包运行环境下，Tauri 启动脚本会在初始化阶段运行 taskkill /F 直接清理可能占用该端口的第三方进程，存在重大系统级冲突。

闭环建议： 落实动态端口分配策略，Tauri 动态拉起 Python 后提取其监听的随机端口号并注入前端的 BaseURL 变量。

P1 级别 对账任务状态流非终态挂起

现象： RPA 加加友任务流中的 SSE 管道在执行结束并更新数据库为终态（如 FAILED/REAL_COMPLETED）后，SSE 链路仍在等待，无法完成自动连接断开，占用后台文件描述符。

闭环建议： 前后端在确认到 `outcome_type` 写入 `success/business/system` 时应当显式发送并捕捉 `EOF`，安全终结该条 SSE 流。

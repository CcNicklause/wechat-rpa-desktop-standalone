# 全项目闭环审计 · 计划

> 任务线：`project-closure-audit`
> 状态：AUDIT_RECONCILED
> 模式：legacy audit / lightweight docs / audit-only
> 本轮约束：先审计、先对账、先验证清单；不先修改业务代码。

## 第一部分 · 需求

### 需求 1：全项目模块闭环审计

- 业务问题：需要确认前端、后端/本地 API、RPA sidecar、状态流、审计日志、任务文档是否一致并形成端到端闭环。
- 审计目标：用统一标准盘点“入口、调用、执行、状态、日志、持久化、配置、文档、测试”是否可追踪、可验证、可恢复。
- 验收标准：输出模块地图、端到端链路、闭环缺口、风险优先级、测试验证清单；本轮不修改业务代码。
- 依据：`docs/tasks/project-closure-audit/state.md` 范围定义，以及本计划当前任务约束。

### 需求 2：为后续 coder-agent 事实流提供审计框架

- 业务问题：后续 `flow.md` 必须反映代码实际行为，不能复述设计；因此需要先定义事实采集口径。
- 审计目标：让 coder-agent 按模块、链路、状态、日志、风险、验证点补充事实，plan-agent 再基于“期望 vs 实际”产出闭环对账。
- 验收标准：本计划提供可直接填写的链路清单、闭环判定标准、风险评分模型、测试验证清单和对账表模板。
- 依据：`docs/tasks/README.md` 定义 plan/flow/state/test 职责与渐进式读取、flow 使用时机。

## 第二部分 · 审计设计

### 2.1 审计原则

- [ ] 事实优先：所有“实际行为”必须由 coder-agent 在 `flow.md` 中依据代码事实补证；本计划只定义审计口径。
- [ ] 闭环优先：每条关键链路必须能从用户入口追踪到 API、执行器、状态变化、日志/事件、持久化与用户反馈。
- [ ] 风险优先：优先识别导致任务不可执行、状态不一致、审计不可追踪、数据丢失、恢复失败的缺口。
- [ ] 不先改代码：本轮只输出计划和后续对账框架，不修改业务代码。
- [ ] 渐进读取：后续事实采集按需要读取代码和对应任务线材料，避免无关全文扫描。

### 2.3 端到端链路清单

> 每条链路需要在后续 `flow.md` 中补充“实际入口、实际文件、实际接口、实际状态、实际日志、实际测试”。

#### 链路 A：桌面端启动到本地服务可用

- [ ] 用户启动桌面应用。
- [ ] 桌面运行时启动或连接本地 API。
- [ ] 动态端口被确定并注入前端 API base。
- [ ] sidecar 进程启动并进入可服务状态。
- [ ] 前端能完成健康检查或首个数据请求。
- [ ] 启动、端口、进程状态有日志或事件可追踪。
- [ ] 失败时用户能看到可理解提示，系统不误判为业务空数据。

#### 链路 B：登录/session 恢复到业务页面可用

- [ ] 用户进入应用后完成登录或 session 恢复。
- [ ] 前端保存并使用必要凭据。
- [ ] 本地 API 或上游调用能识别当前身份/终端。
- [ ] token/session 过期时有明确处理。
- [ ] 登录状态、终端状态、页面访问权限一致。
- [ ] 敏感信息不在日志、URL、非必要持久化中明文暴露。

#### 链路 C：线索/客户列表加载到详情展示

- [ ] 前端请求线索/客户列表。
- [ ] 本地 API 或本地缓存返回真实数据。
- [ ] 列表状态、详情状态、KPI/统计口径一致。
- [ ] 空列表、加载失败、部分字段缺失有明确 UI。
- [ ] 用户打开详情后能看到状态、步骤、审计记录和可执行动作。

#### 链路 D：发起 RPA 加微/任务执行

- [ ] 用户从前端触发加微或相关任务。
- [ ] 前端提交完整且校验过的请求。
- [ ] 本地 API 创建任务并返回任务标识。
- [ ] sidecar 接收任务，执行微信识别、搜索、添加、备注等步骤。
- [ ] 每一步写入状态、失败原因、截图/证据或可关联日志。
- [ ] 前端实时或准实时展示步骤进度。
- [ ] 成功、失败、取消、重试后的最终状态可解释且可追踪。

#### 链路 E：状态同步与审计事件传播

- [ ] 状态变化在 sidecar、本地 API、前端展示之间保持一致。
- [ ] SSE/轮询/刷新策略能覆盖执行中与完成后状态。
- [ ] 审计事件包含任务 ID、线索 ID、步骤、状态、时间、错误码、trace 关联字段。
- [ ] UI 中的审计记录与底层日志/事件可对账。
- [ ] 重启应用后状态不会回退、丢失或误显示。

#### 链路 F：失败恢复、重试与人工介入

- [ ] 可区分依赖失败、微信窗口异常、识别失败、网络失败、上游失败、数据校验失败。
- [ ] 可重试失败与不可重试失败有明确边界。
- [ ] 重试不会造成重复添加、重复上报、状态覆盖或审计断链。
- [ ] 需要人工介入时 UI 有明确提示和下一步建议。
- [ ] 恢复动作和结果被记录到审计日志。

#### 链路 G：外部上报/终端状态闭环

- [ ] 终端 record/heartbeat/status 或相关上报接口调用时机清晰。
- [ ] 上报失败有重试、降级或可见告警。
- [ ] 本地任务状态与上游状态字段映射一致。
- [ ] 外部错误码映射到本地可理解原因。
- [ ] 上报结果可从日志或审计事件追踪。

#### 链路 H：任务文档到代码事实闭环

- [ ] `docs/tasks/README.md` 中任务线状态与各任务线 `state.md` 一致。
- [ ] 当前任务线 `plan.md` 的期望与 `flow.md` 的事实可逐项对账。
- [ ] 历史 DONE/IMPLEMENTED 任务线有测试结果或明确残留项。
- [ ] 文档中的状态、接口、枚举、测试命令未与代码事实冲突。

### 2.4 闭环判定标准

每条模块或链路按以下标准判定：

| 判定项 | PASS | PARTIAL | FAIL | BLOCKED |
|---|---|---|---|---|
| 入口可达 | 用户或系统入口明确且可触发 | 入口存在但条件/权限/路径不清 | 找不到可用入口 | 需要运行环境或外部账号才能判断 |
| 调用连续 | 前端/API/sidecar 调用链完整 | 存在分支未确认或异常路径缺失 | 调用链断裂或参数不匹配 | 缺少依赖无法确认 |
| 状态一致 | 枚举、转换、展示一致 | 主路径一致但边界状态缺失 | 前后端或文档状态冲突 | 缺少事实材料 |
| 审计可追踪 | 有 trace/id/时间/结果/错误原因 | 只有部分字段或部分步骤有记录 | 无法关联用户动作与执行结果 | 日志需运行后采集 |
| 错误可恢复 | 错误分类、提示、重试/人工处理明确 | 仅部分错误可恢复 | 失败后无可操作路径或易重复执行 | 需真实依赖验证 |
| 数据持久 | 重启/刷新后关键数据不丢失 | 部分缓存依赖内存或刷新策略不明 | 状态或审计记录丢失 | 需数据库/本地存储检查 |
| 文档一致 | 文档、代码事实、测试结果一致 | 存在小范围过期说明 | 文档误导实际行为 | 缺少对应文档 |
| 测试可证 | 有自动化或手工验证路径 | 只有部分链路有验证 | 无验证方式 | 环境不可用 |

#### 全局闭环通过标准

- [ ] P0 链路 A、D、E、F 至少达到 PASS；如因环境不可验证，必须标注 BLOCKED 和所需条件。
- [ ] 所有 P0 风险无 FAIL；如存在 FAIL，后续对账必须进入 `STATUS: NEEDS_ITERATION`。
- [ ] P1 链路至少无不可解释 FAIL；允许 PARTIAL，但必须有明确后续补证或修复建议。
- [ ] 关键状态枚举、错误码、审计事件字段至少完成一轮对账。
- [ ] 测试验证清单覆盖主路径、边界、错误、集成和性能/稳定性基线。

### 2.6 风险评分模型

#### 评分公式

- 严重度 S：1-5。5 表示导致核心业务不可用、数据丢失、安全泄露或状态不可恢复。
- 发生概率 P：1-5。5 表示主路径高频触发或默认配置下必现。
- 可发现性 D：1-5。5 表示用户/测试难以及时发现，或发现时已造成后果。
- 影响范围 B：1-3。3 表示跨前端、本地 API、sidecar、外部集成或文档多域扩散。
- 风险分 R = S × P × D + B。

#### 风险等级

| 等级 | 分数 | 处理策略 |
|---|---:|---|
| P0 必改 | R >= 65，或触发强制 P0 条件 | 后续对账必须列入必改，未解决不得收敛 |
| P1 建议改 | 35 <= R < 65 | 进入迭代计划，允许有明确缓解方案后阶段性收敛 |
| P2 可选 | R < 35 | 记录为优化项或技术债 |

#### 强制 P0 条件

- [ ] 主链路无法发起或无法完成 RPA 任务。
- [ ] 状态显示与实际执行结果冲突，可能误导用户继续操作。
- [ ] 失败后不可恢复且可能重复执行、重复添加或重复上报。
- [ ] 审计日志无法关联关键动作与执行结果。
- [ ] session/token/本地 API 存在明显敏感信息暴露或未授权访问风险。
- [ ] 重启/刷新后关键任务状态或审计证据丢失。

## 第三部分 · 测试验证清单

> 本清单用于指导后续 test-agent 或人工验证。当前计划阶段不执行测试，不修改代码。

### 3.1 功能正确性

- [ ] 桌面端冷启动后，本地 API 可用，前端能加载首屏数据。
- [ ] 动态端口变化后，前端 API base 指向正确本地服务。
- [ ] 登录或 session 恢复后，用户能访问业务页面。
- [ ] 线索/客户列表、详情、KPI/统计数据口径一致。
- [ ] 用户发起 RPA 任务后，任务 ID、步骤进度、最终状态可见。
- [ ] 成功任务的审计记录包含发起、执行、完成信息。
- [ ] 失败任务的审计记录包含失败步骤、错误码/原因和下一步建议。
- [ ] 终端 record/heartbeat/status 或相关上报在约定时机触发。

### 3.2 边界条件

- [ ] 首次启动无配置、无缓存、无历史 session。
- [ ] 本地端口被占用或动态端口变化。
- [ ] sidecar 启动慢、启动失败、运行中退出。
- [ ] 线索列表为空、字段缺失、字段类型异常。
- [ ] 重复点击发起任务按钮或并发发起多个任务。
- [ ] 任务执行中刷新页面、关闭重开桌面端。
- [ ] 同一线索重复执行、重试执行、跨状态执行。
- [ ] 审计日志数量大、事件乱序、SSE/轮询中断。

### 3.3 错误处理

- [ ] 本地 API 不可达时，前端显示连接/服务异常，而不是误显示业务空态。

# 桌面客户端登录对接文档

> **目标读者**：另一个仓库的 RPA 自动加微桌面项目（Tauri + Python 引擎 + React 前端）
> **核心诉求**：与 Web Portal 共用同一套账号体系，桌面端**只登录不注册**，注册仍在 Web 完成
> **后端服务**：aisales-portal API（NestJS，本仓库 `api/`）
> **文档版本**：v1（2026-06-28）

---

## 1. 总体架构与账号模型

```
┌─────────────────────┐         ┌─────────────────────┐
│  Web Portal (React) │         │  Desktop (Tauri)    │
│  pages/Login.tsx    │         │  React + Python引擎 │
└──────────┬──────────┘         └──────────┬──────────┘
           │                               │
           │  POST /api/v1/auth/login      │
           │  POST /api/v1/auth/login-by-sms
           │  POST /api/v1/sms/send-code   │
           ▼                               ▼
   ┌─────────────────────────────────────────────────┐
   │   aisales-portal API (NestJS)                   │
   │   - AuthController  : 登录 / 重置 / getMe      │
   │   - SmsController   : 发短信验证码             │
   │   - JwtStrategy     : Bearer Token 校验        │
   │   - MgrController   : 转发到 MGR 网关（含机器码）│
   └────────────────┬────────────────────────────────┘
                    │
                    ▼
           ┌──────────────────┐
           │ MGR 网关          │
           │ rpa-terminal/*   │   ← 桌面端注册 / 心跳 / 列表
           └──────────────────┘
```

**关键点**：
- **同一个 users 表，同一个 JWT 签发逻辑**。桌面端登录拿到的 token 与 Web 登录拿到的 token 完全等价，都能访问 `/api/v1/*` 受保护接口。
- **桌面端不开放注册入口**。`POST /auth/register` 接口仍然存在（验证码注册），但你的客户端 UI 应当**不暴露**，提示用户去 Web 完成注册。
- **机器码 / 终端绑定**走的是另一条链路：`/api/v1/mgr/rpa-terminal/*`，与登录解耦。登录成功后再上报终端硬件信息。

## 2. 登录可选方案

后端目前 ([api/src/auth/auth.controller.ts](api/src/auth/auth.controller.ts)) 提供 4 个 `@Public()` 接口：

| 接口 | 说明 | 桌面端是否使用 |
|---|---|---|
| `POST /api/v1/auth/login` | 手机号 + 密码登录 | ✅ **主入口** |
| `POST /api/v1/auth/login-by-sms` | 手机号 + 短信验证码登录 | ✅ 推荐作为备选 |
| `POST /api/v1/auth/register` | 手机号 + 密码 + 短信码 注册 | ❌ 桌面端不暴露 |
| `POST /api/v1/auth/reset-password` | 手机号 + 短信码 重置密码 | ⚠️ 可选，建议引导去 Web |

**推荐做法**：桌面端实现「密码登录」和「验证码登录」两个 Tab，与 Web 一致 ([web/src/pages/Login.tsx](web/src/pages/Login.tsx))。**不提供注册入口**，只在登录失败提示中加一行「还没有账号？请前往 https://aisales.weimob.com 注册」。

## 3. API 详解

> 全部接口走 `/api/v1` 前缀（见 [api/src/main.ts:42](api/src/main.ts#L42)）。
> Content-Type 一律 `application/json`。

### 3.1 发送短信验证码

```http
POST /api/v1/sms/send-code
Content-Type: application/json

{
  "phone": "13800138000",
  "type": "login"            // register | login | reset，桌面端固定传 "login"
}
```

**响应**：
```json
{ "success": true, "message": "验证码已发送" }
```
> Mock / QA 环境会额外返回 `"code": "1111"`（详见 [api/src/sms/sms.controller.ts:33](api/src/sms/sms.controller.ts#L33) 与 [api/src/sms/sms.service.ts:153](api/src/sms/sms.service.ts#L153)），生产环境不返回。

**限流规则**（[api/src/sms/sms.service.ts:18-20](api/src/sms/sms.service.ts#L18-L20)）：
- 同一手机号 **60 秒**内只能发送 1 次（前端需要倒计时）
- 每日上限 **10 次**
- 验证码有效期 **5 分钟**

### 3.2 密码登录

```http
POST /api/v1/auth/login
Content-Type: application/json

{
  "phone": "13800138000",
  "password": "your_password"     // 至少 6 位
}
```

校验规则（[api/src/auth/auth.controller.ts:8-16](api/src/auth/auth.controller.ts#L8-L16)）：
- 手机号正则 `/^1[3-9]\d{9}$/`
- 密码 `minLength: 6`

### 3.3 验证码登录

```http
POST /api/v1/auth/login-by-sms
Content-Type: application/json

{
  "phone": "13800138000",
  "code": "1234"                  // 至少 4 位
}
```

> **该手机号未注册时返回 `401 该手机号未注册`**（[auth.service.ts:65](api/src/auth/auth.service.ts#L65)），桌面端要把这个错误提示替换成「请先前往 Web 注册」。

### 3.4 登录成功响应（两个登录接口结构一致）

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9....",
  "user": {
    "id": 12,
    "user_id": "u_xxxxxxxx",
    "email": null,
    "phone": "13800138000",
    "name": "测试用户",
    "role": "sales",
    "tenant_id": 100024
  }
}
```

JWT 负载（[auth.service.ts:149-154](api/src/auth/auth.service.ts#L149-L154)）：
```ts
{
  sub: user.user_id,          // 用户唯一 ID（字符串）
  tenant_id: number,          // 租户/企业 ID（数字！）
  phone: string,
  role: string,
  iat, exp                    // exp = iat + JWT_EXPIRES_IN（默认 24h）
}
```

> 重要：`tenant_id` 必须保持**数字类型**。后端 `JwtStrategy.validate()` 会严格校验类型不对就抛 401（[jwt.strategy.ts:31-43](api/src/auth/jwt.strategy.ts#L31-L43)）。桌面端如果自己解析 JWT 显示用户信息，注意不要把它字符串化。

### 3.5 获取当前用户（验证 token 是否仍有效）

```http
GET /api/v1/auth/me
Authorization: Bearer <access_token>
```

返回的字段同上面 `user`。**桌面端启动时调用一次该接口判断 token 是否过期**，比单纯的本地过期戳更可靠。

### 3.6 错误响应统一格式

NestJS 默认格式：
```json
{
  "statusCode": 401,
  "message": "手机号或密码错误",
  "error": "Unauthorized"
}
```
- `401`：用户名/密码错、验证码错、token 失效、token 缺失
- `400`：参数校验失败、短信发送过于频繁、当日上限
- `500`：服务端异常

## 4. JWT / Bearer Token 认证机制

### 4.1 携带方式

所有需要登录后才能访问的接口都需要在请求头加：

```http
Authorization: Bearer <access_token>
```

服务端的 JWT 策略只接受 `Bearer` 前缀（[jwt.strategy.ts:24](api/src/auth/jwt.strategy.ts#L24)）。

### 4.2 Token 有效期

- 默认 **24h**，可由环境变量 `JWT_EXPIRES_IN` 调整（[环境变量配置.md](docs/环境变量配置.md#L37-L42)）
- 服务端**不提供刷新 token 接口**，过期需要重新登录
- 推荐桌面端策略：
  1. 启动 / 切前台时调一次 `GET /auth/me`，401 就跳登录页
  2. 业务请求 401 时自动跳登录页
  3. 不要在本地解析 `exp` 提前续期，因为没有 refresh 接口

### 4.3 双认证（可选高阶用法）

后端还支持 **APIKey** 认证作为替代（`x-api-key` header 或 `Bearer <api_key>`，[combined-auth.guard.ts](api/src/auth/combined-auth.guard.ts)），用于服务端到服务端调用。桌面端**不应使用** APIKey 登录，因为它绑定到用户而非设备，泄漏风险更高。

## 5. 跨域（CORS）问题

### 5.1 后端 CORS 配置

[api/src/main.ts:36-39](api/src/main.ts#L36-L39)：

```ts
app.enableCors({
  origin: runtimeConfig.FRONTEND_URL,   // 单值白名单！
  credentials: true,
});
```

`FRONTEND_URL` 默认 `http://localhost:5190`，由环境变量配置（[环境变量配置.md L21](docs/环境变量配置.md#L21)）。

### 5.2 Tauri 端面临的问题

Tauri WebView 加载本地资源时，`Origin` 通常是：
- Tauri v1 (Windows): `https://tauri.localhost`
- Tauri v2 (Windows): `http://tauri.localhost`
- 开发模式 (dev server): `http://localhost:1420` 或类似端口

**与后端 `FRONTEND_URL=http://localhost:5190` 不匹配，会被 CORS 拦截。**

### 5.3 三种解决方案（推荐顺序）

#### ✅ 方案 A（推荐）：在 Tauri 端用 Rust HTTP 客户端发起请求

通过 `tauri::command` 把 HTTP 请求转到 Rust 后端发出，**绕过浏览器 CORS**：

```rust
// src-tauri/src/api.rs
#[tauri::command]
async fn login(phone: String, password: String) -> Result<serde_json::Value, String> {
    let client = reqwest::Client::new();
    let resp = client
        .post("https://api.aisales.weimob.com/api/v1/auth/login")
        .json(&serde_json::json!({ "phone": phone, "password": password }))
        .send().await.map_err(|e| e.to_string())?;
    resp.json().await.map_err(|e| e.to_string())
}
```

React 端：
```ts
import { invoke } from '@tauri-apps/api/core';
const result = await invoke('login', { phone, password });
```

**优点**：完全规避 CORS、Cookie、混合内容；token 也可直接存到 Rust 端（更安全，见第 7 节）。

#### ✅ 方案 B：后端 CORS 白名单加上 Tauri Origin

在部署 Portal 时把 `FRONTEND_URL` 改成支持多源（需要后端改一行）：

当前 [main.ts:37](api/src/main.ts#L37) 只接受**单值** `origin`，需要改为：

```ts
app.enableCors({
  origin: (origin, cb) => {
    const allow = [
      runtimeConfig.FRONTEND_URL,
      'http://tauri.localhost',
      'https://tauri.localhost',
      'http://localhost:1420',
    ];
    if (!origin || allow.includes(origin)) cb(null, true);
    else cb(new Error('Not allowed by CORS'));
  },
  credentials: true,
});
```

> ⚠️ 改后端要走 `/orchestrate legacy` 工作流；如果你不想改后端，直接走方案 A。

#### ⚠️ 方案 C：Tauri 端禁用 WebView 同源策略

不推荐，破坏所有跨域防御，仅适合无网络敏感数据的内部 demo。

### 5.4 Python 引擎到后端的调用

桌面端的 Python 自动加微引擎如果要直接调 Portal API（例如查 RPA 终端状态），**不存在跨域问题**（Python 不是浏览器环境），只要带 `Authorization: Bearer <token>` 即可。

```python
import requests

resp = requests.post(
    "https://api.aisales.weimob.com/api/v1/mgr/rpa-terminal/list",
    json={"pageNum": 1, "pageSize": 10},
    headers={"Authorization": f"Bearer {access_token}"},
    timeout=10,
)
```

> Tauri 前端和 Python 引擎要**共享 token**：建议 Tauri Rust 把 token 写到一个文件（如 `%APPDATA%\aisales\session.json`），Python 启动时读取；或者 Rust 通过 stdin 注入。

## 6. 机器码 / 终端绑定（RPA 桌面专有）

### 6.1 概念辨析

后端**没有把机器码作为登录凭证**。机器码的作用是：
1. 把"哪台桌面在跑哪个微信号"和"哪个用户"关联起来
2. 上报终端硬件信息、心跳、状态供 Web Portal 展示（[web/src/pages/Settings/RpaClientManagement.tsx](web/src/pages/Settings/RpaClientManagement.tsx)）

所以**机器码上报是登录之后的事情**。

### 6.2 已有接口

桌面端登录拿到 JWT 后，所有 `/mgr/*` 请求会被 [api/src/mgr/mgr.controller.ts](api/src/mgr/mgr.controller.ts) 转发到下游 MGR 网关，并自动注入 `orgId = tenant_id`（[mgr.service.ts:73-77](api/src/mgr/mgr.service.ts#L73-L77)）。

桌面端关心的接口：

| 接口 | 用途 | 何时调用 |
|---|---|---|
| `POST /api/v1/mgr/rpa-terminal/register` | 终端首次注册（上报硬件信息） | 客户端**首次启动** |
| `POST /api/v1/mgr/rpa-terminal/heartbeat` | 心跳上报 | **每 30s** 一次 |
| `POST /api/v1/mgr/rpa-terminal/list` | 查询当前租户下所有终端 | Web 端 / 调试用 |

> 接口的字段以 MGR 网关的 OpenAPI 为准，本仓库的 [client.ts:585-625](web/src/api/client.ts#L585-L625) 给出了 `RpaTerminal` 的完整类型，可以照搬。

### 6.3 机器码（terminalId）生成建议

后端字段叫 **`terminalId`**（[client.ts:596](web/src/api/client.ts#L596)），用作设备唯一标识。生成策略：

1. **优先用 MAC + 主板序列号 hash**（稳定，重装系统不变）
   ```rust
   // Tauri Rust
   use sha2::{Sha256, Digest};
   fn machine_id() -> String {
       let mac = mac_address::get_mac_address().unwrap().unwrap().to_string();
       let board = wmi_get_board_serial();   // 调 WMIC 或 wmi crate
       let mut h = Sha256::new();
       h.update(format!("{}|{}", mac, board));
       format!("{:x}", h.finalize())[..32].to_string()
   }
   ```
2. **退而求其次用 Windows MachineGuid**：
   `HKLM\SOFTWARE\Microsoft\Cryptography\MachineGuid`（重装系统会变）
3. **本地存一份**：算出来后写到 `%APPDATA%\aisales\terminal.id`，避免每次重算

注册时把硬件信息一并上报（字段定义见 [client.ts:593-613](web/src/api/client.ts#L593-L613)）：
```json
{
  "terminalId": "<sha256>",
  "name": "DESKTOP-XXX",                // 计算机名
  "type": "windows",
  "ipAddress": "192.168.1.x",
  "macAddress": "AA:BB:CC:DD:EE:FF",
  "osName": "Windows 10 Pro",
  "osVersion": "10.0.19045",
  "cpuInfo": "Intel i7-...",
  "memoryGb": 16,
  "diskGb": 512,
  "screenResolution": "1920x1080"
}
```

### 6.4 一个用户多台桌面

后端模型是 **一个 tenant_id 下挂多个 terminal**（[client.ts:594-595](web/src/api/client.ts#L594-L595)），所以同一账号可以在多台桌面同时登录，每台机器自己的 `terminalId` 不同。在 Web 的 RPA 客户端管理页可以看到所有终端列表。

## 7. 桌面端登录流程（推荐实现）

```
┌─ App 启动 ──────────────────────────────────────────────┐
│ 1. Rust 端读取 %APPDATA%\aisales\session.json          │
│    └─ 没有 / token 过期 → 进入登录页                   │
│    └─ 有 token → 调 GET /auth/me 校验                  │
│       └─ 200 → 进入主界面                              │
│       └─ 401 → 清掉本地 session，进登录页              │
└──────────────────────────────────────────────────────┘
        │
        ▼
┌─ 登录页（React，调 invoke） ──────────────────────────┐
│  用户输入 手机号 + 密码（或 验证码）                  │
│  └─ invoke('login', {...})                            │
│     └─ Rust: POST /api/v1/auth/login → token + user   │
│     └─ Rust: 写 session.json（含 token、user 信息）   │
│  └─ 跳主界面                                          │
└──────────────────────────────────────────────────────┘
        │
        ▼
┌─ 主界面初始化 ──────────────────────────────────────┐
│ 1. 算/读 terminalId                                  │
│ 2. POST /api/v1/mgr/rpa-terminal/register            │
│    （带 Authorization: Bearer <token>）              │
│ 3. 启动 30s 心跳定时器                                │
│ 4. 把 token 通过 stdin 或环境变量传给 Python 引擎    │
└────────────────────────────────────────────────────┘
```

### 7.1 Token 存储位置（按安全程度排序）

1. **最佳**：Tauri Rust 端用 [`tauri-plugin-store`](https://github.com/tauri-apps/plugins-workspace) 或操作系统密钥管理（`keyring` crate）。
2. **可接受**：写入 `%APPDATA%\aisales\session.json`，文件 ACL 设为只允许当前用户读。
3. **不推荐**：localStorage / sessionStorage（WebView 进程崩溃可能被注入读取）。

### 7.2 Python 引擎共享 token

```rust
// Rust 启动 Python 子进程时注入
let mut cmd = Command::new("python");
cmd.arg("engine.py");
cmd.env("AISALES_TOKEN", &token);
cmd.env("AISALES_API_BASE", "https://api.aisales.weimob.com/api/v1");
cmd.spawn()?;
```

```python
# engine.py
import os, requests
TOKEN = os.environ["AISALES_TOKEN"]
API   = os.environ["AISALES_API_BASE"]
session = requests.Session()
session.headers["Authorization"] = f"Bearer {TOKEN}"
```

> Token 过期时 Python 端收到 401，应该**通知 Tauri 端**（IPC / 写 stderr 特殊标记）触发重新登录，**不要在 Python 里弹登录窗**。

## 8. 安全要点

1. **不要在桌面端硬编码任何"超级密码"**。当前后端有一个开发用万能密码 `"111111"`（[auth.service.ts:37](api/src/auth/auth.service.ts#L37)）和万能验证码 `"1111"`（[sms.service.ts:153](api/src/sms/sms.service.ts#L153)），生产环境**必须**通过环境变量或代码移除——这是当前 portal 仓库需要清理的安全债务，桌面端**不能依赖**这两个值。
2. **JWT_SECRET 必须与 Portal 共享同一个密钥**（同一 Portal 实例就行）。如果你部署了多个 Portal 节点，确保它们的 `JWT_SECRET` 一致，否则同一个 token 在不同节点验证结果不同。
3. **HTTPS 必选**。生产环境桌面端**只**走 `https://`，不要给用户留 HTTP 选项——`Bearer <token>` 在 HTTP 链路上等于明文密码。
4. **terminalId 算法要做防伪**。如果用户能轻易伪造另一台机器的 terminalId，会绕过 RPA 终端绑定限制。可以让服务端额外签发一个 `terminal_token`（注册时下发），后续上报必须带。

## 9. 与 Web 共用账号的注意事项

| 场景 | 行为 | 备注 |
|---|---|---|
| 同一手机号在 Web 注册 → 桌面登录 | ✅ 直接登录 | 用同一个 user 表 |
| Web 改了密码 → 桌面老 token | ⏳ 老 token 仍有效到过期 | 后端没有 token 黑名单 |
| 桌面注销 | 仅本地清 token | 不调后端（没有 logout 接口） |
| 同一账号同时在 Web 和桌面登录 | ✅ 并行有效 | JWT 无状态，互不影响 |
| Web 端忘记密码 → 桌面端是否也要做 | ❌ 不建议做 | 引导到 Web，减少桌面端短信用量 |
